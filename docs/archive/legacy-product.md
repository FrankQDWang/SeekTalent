# Product

## Register

product

## Users

SeekTalent is for internal recruiting and headhunting colleagues who process many JD sessions each day. They work from a local/LAN web app, use source-specific recruiting accounts such as CTS and Liepin, and need one accountable workspace per JD plus optional notes.

## Product Purpose

SeekTalent turns one JD session into a multi-source recruiter workbench: clarify requirements, run source-specific search, observe agent progress, review candidates, and preserve evidence without leaking raw resume data into ordinary UI, memory, logs, or events. Success means a recruiter can move faster through must-have triage, source search, candidate review, and detail-open decisions while keeping source attribution and audit state recoverable.

## Brand Personality

Calm, expert, operational. The interface should feel like a focused internal agent console for serious recruiting work, not like a SaaS marketing dashboard or a generic admin panel.

## Anti-references

Do not make this look like a landing page, CRM dashboard, decorative card grid, or generic beige SaaS app. Avoid hero sections, oversized marketing typography, nested card layouts, ornamental gradients, and any UI that hides the live agent workflow behind static forms.

## Design Principles

- The JD session is the workspace. Every screen should make the current JD, sources, progress, and review queue feel connected.
- Show the agent at work. Progress belongs in a visible strategy canvas, log lane, source cards, and bottom timeline, not only in toast messages or tables.
- Evidence stays attributed. Candidate results must show source badges and evidence level while keeping raw provider material behind protected boundaries.
- Reference-page structure is the product model. The supplied recruiter-agent HTML defines the baseline topology: JD/source panel, center strategy canvas, right log/candidate lane, bottom timeline, plus an added session rail.
- Internal-tool density over decoration. Recruiters should scan, compare, approve, reject, pause, and resume without fighting layout.

## Accessibility & Inclusion

Target WCAG AA for text contrast, keyboard navigation, focus states, and form labeling. Motion should convey run state but respect reduced-motion preferences. Dense Chinese and English mixed content must wrap predictably without overlap.
