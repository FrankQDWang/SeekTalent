# OpenCLI 1.8.6 薄 Fork 的最小浏览器契约

日期：2026-07-13
结论基线：OpenCLI tag `v1.8.6`，commit `cad35e7a6a5ff3f7d6b859bfa4c45195c0390260`

## 结论

不需要把 SeekTalent 的 run、猎聘判断、锁定层或倒计时 UI 写进 OpenCLI。最小 fork 只补三项通用能力：

1. 只读查找符合 URL 条件的现有用户 tab，并以选定 tab 作为宿主窗口锚点；
2. 在锚点所在窗口创建 `active: false` 的新 tab，同时只拥有这个新 tab，不拥有、不绑定锚点 tab，也不拥有宿主窗口；
3. 按精确 tab 身份真实关闭，验证 tab 已消失，并返回可区分 `closed`、`already_missing`、`failed` 的结果。

SeekTalent 对每个 owned tab 使用一个唯一、不透明的 OpenCLI session。这样可以直接复用 `1.8.6` 已有的“一个 session 对应一个 `preferredTabId`”以及持久化 idle alarm，不需要把 OpenCLI 改成“一 session 管一组 tab”。同一 session 不能连续创建多个 tab。

60 秒是空闲时间，不是 tab 从创建开始计算的硬寿命。每条命令完成后重新开始 60 秒；source run 完成时立即请求关闭。关闭失败只记录为 cleanup failure，不能覆盖已经成功的业务结果，也不能把整个 run 卡住。

## 上游 1.8.6 已经具备什么

以下判断来自上游 tag，而不是 SeekTalent 文档：

- [`TargetLease`](https://github.com/jackwener/OpenCLI/blob/v1.8.6/extension/src/background.ts) 每个 session 只保存一个 `preferredTabId`，并把 lease、`idleDeadlineAt` 和窗口信息写入 `chrome.storage.local`。
- `resetWindowIdleTimer` 同时使用 `setTimeout` 和 `chrome.alarms`；service worker 重启后，`reconcileTargetLeaseRegistry` 会按原绝对 deadline 恢复，不会无条件赠送新的完整周期。
- `handleCommand` 接受单位为秒的 `idleTimeout`，内部乘以 `1000`；命令执行期间不会释放 lease，命令完成后重新计时。因此 SeekTalent 应传 `60`，不是 `60000`。
- 用户手动关闭 tab 时，现有 `chrome.tabs.onRemoved` 路径会移除对应 lease。
- CLI 已支持用明确的 page/target 身份执行 page 命令；[`browser eval`](https://github.com/jackwener/OpenCLI/blob/v1.8.6/src/cli.ts) 也支持 `--tab <targetId>`。SeekTalent 可以用固定、受限的方法注入灰色锁定层，无需给 fork 增加猎聘 UI。

## 上游行为为什么不能直接使用

### 新建 tab 会新建或复用 OpenCLI 自有窗口

[`ensureOwnedContainerWindow`](https://github.com/jackwener/OpenCLI/blob/v1.8.6/extension/src/background.ts) 在找不到 OpenCLI owned container 时调用 `chrome.windows.create`。`background` 只把 `focused` 设为 `false`，并不表示“使用用户现有窗口”。`createOwnedTabLease` 和 `tabs new` 又固定使用 `active: true`。

### 最后一个 tab 不会被真实关闭

同文件的 `releaseLease` 在窗口内没有其他 owned lease 时把 tab 导航到 `about:blank`，并设置 `active: true`。只有导航失败才尝试 `chrome.tabs.remove`，而 remove 错误被吞掉。随后 lease 仍被删除，所以调用方无法知道 tab 是否真的消失。

### owned tab 与 owned window 被混在一起

当前 `owned: true` 实际隐含“tab 位于 OpenCLI owned container”。恢复时会登记 owned window、创建 tab group，导航后发现 tab 换窗还会尝试把它移回去。对于“用户窗口里的 SeekTalent-owned tab”，这些操作都不安全：tab 可以归 SeekTalent，但窗口永远归用户。

### 版本握手不能证明 fork 能力

扩展 hello 目前只发送扩展版本和 CLI 兼容范围；[`daemon`](https://github.com/jackwener/OpenCLI/blob/v1.8.6/src/daemon.ts) 也只记录这些字段。一个版本号看似兼容的上游扩展仍可能被误当成具备 fork 新能力。

## 建议的通用协议

字段名可以在原型中微调，但语义必须保持如下边界。

### 1. 查找宿主候选

新增只读 tab 操作，例如：

```json
{
  "action": "tabs",
  "op": "find",
  "urlPrefix": "https://h.liepin.com/"
}
```

返回所有匹配候选的稳定 page/target 身份、URL、是否 active、是否位于最近聚焦窗口等选择信息。该操作不得绑定、激活、导航或声明拥有任何用户 tab。URL 规则由 SeekTalent 的猎聘适配层给出；OpenCLI 只提供通用、安全的 `http/https` 匹配能力。

没有匹配时返回稳定错误 `host_tab_not_found`。这表示当前 Chrome context 中没有可用的既有猎聘会话；SeekTalent 直接提示用户先登录 `h.liepin.com`，不创建窗口、不自动登录。

多个候选如何选择由独立事项 [确定多个猎聘窗口或登录 Tab 的宿主选择规则](https://github.com/FrankQDWang/SeekTalent/issues/291) 决定，不应硬编码进 fork。

### 2. 在用户窗口创建 owned tab

扩展 `tabs new` 或新增等价操作，接受锚点 page 和显式 active 标志：

```json
{
  "action": "tabs",
  "op": "new",
  "hostPage": "<existing-user-tab-target-id>",
  "url": "https://h.liepin.com/",
  "active": false,
  "session": "<opaque-tab-session>"
}
```

扩展再次确认锚点仍存在，然后调用 `chrome.tabs.create({ windowId: anchor.windowId, active: false, ... })`。创建结果形成新的 owned lease，但需新增并持久化明确的窗口所有权，例如 `windowOwnership: "borrowed"`：

- `ownership: "owned"` 表示新 tab 可关闭；
- `windowOwnership: "borrowed"` 表示只能把该窗口作为位置，绝不能关闭、聚组、聚焦或声明为 owned container；
- 锚点 user tab 始终不是 lease 的 `preferredTabId`，也不得被 bind；
- tab 被用户拖到其他窗口后，ownership 跟随 tab，更新记录的 `windowId`，不得强行移回原窗口。

返回值至少包含新 tab 的 page/target 身份、`active: false` 和 placement 类型。创建失败使用稳定错误，例如 `host_page_missing`、`host_window_unavailable`、`tab_create_failed`。

### 3. 精确且可验证的关闭

关闭只接受 owned session 的当前 page/target 身份，不接受易漂移的 tab index。对 borrowed host window 中的 owned tab，`releaseLease` 必须：

1. detach CDP；
2. 对精确 tab ID 调用 `chrome.tabs.remove`；
3. 通过 `tabs.onRemoved` 或再次 `chrome.tabs.get` 验证 tab 已不存在；
4. 只有在 `closed` 或 `already_missing` 后才删除 ownership 记录；
5. `failed` 时保留足够的 lease/deadline 状态供后台恢复重试，不得退化为 `about:blank`。

建议结果：

```json
{
  "requested": "<target-id>",
  "outcome": "closed | already_missing | failed",
  "verified": true,
  "errorCode": null
}
```

`closed` 与 `already_missing` 都是幂等成功。`failed` 必须保留稳定错误码。SeekTalent 收到 `failed` 后记录 cleanup 告警并结束 run；后台重试和 orphan 处理的完整策略由 [设计 source run 到 owned tab 的生命周期与崩溃恢复模型](https://github.com/FrankQDWang/SeekTalent/issues/289) 继续决定。

## 一 tab 一 session

SeekTalent 为每个 tab 生成新 session，例如由 source run ID 与随机 tab ID 组成，但 OpenCLI 必须把它当作不透明字符串。run 内维护 `owned_tabs: session -> page` 清单：

- 新开一个 tab，就新增一个 session；
- 所有命令显式携带对应 session 和 page；
- 命令的 `idleTimeout` 固定传 `60` 秒；
- run 完成时遍历清单并请求立即关闭；
- 某个关闭失败不会中断其余 tab 的关闭，也不会改变业务结果。

这不是“最多两个 tab”。run 可以按真实工作需要创建任意数量，但每个 tab 都有独立 ownership、deadline 和回收结果。

## 灰色锁定层留在 SeekTalent

锁定层、页面底部的 60 秒倒计时和“两侧细线”都属于 SeekTalent 产品行为，不进入 OpenCLI fork。建议在 SeekTalent wrapper 暴露一个固定的 `install_controlled_tab_lock(page, deadline)` 方法，内部使用现有定向 eval；不要把任意 eval 开放给业务层。

页面导航会清除 DOM 中的锁定层，所以每次导航完成后重新注入。倒计时在页面内根据绝对 deadline 自行刷新，不能每秒向 OpenCLI 发命令，否则会不断重置 idle deadline。用户仍可通过 Chrome 自身的关闭按钮关闭 tab。

- `src/seektalent/opencli_launcher.py`：开发期 fork 选择；生产期不得联网安装；

- `src/seektalent/opencli_browser/automation.py`：定向命令、60 秒 timeout、固定锁定层入口；

## 兼容与安全约束

- 保留上游 owned-container 路径供其他 OpenCLI 使用者使用；SeekTalent 只走新增的 borrowed-host 路径。
- 新 registry 需要版本升级并兼容读取 v2；不能把旧 owned-container lease 猜成 borrowed-host lease。
- borrowed user tab 与 borrowed host window 中的 owned tab 是两种不同状态。前者永远不能关闭，后者只允许关闭精确 tab。
- 所有命令继续用 page/target 身份，不用 tab index，也不依赖当前 active tab。
- host 查找和新建仅允许 `http/https` URL；SeekTalent 再把范围收紧到 `https://h.liepin.com/`。
- fork hello 应公布稳定 capability 名称，例如 `tab.find.v1`、`tab.create-in-existing-window.v1`、`tab.close-verified.v1`。缺任一能力时启动前明确报错，不能静默退回会新建窗口的上游路径。

## 离线生产分发约束

生产用户没有可依赖的海外网络。SeekTalent 安装包必须把 fork 的 CLI/daemon 和配套 Chrome 扩展作为同一版本的本地资产成对内置；生产运行时不得执行 `npm install`，也不得访问 npm 或 GitHub。

当前 [`src/seektalent/opencli_launcher.py`](../../src/seektalent/opencli_launcher.py) 固定安装 `@jackwener/opencli@1.8.6`，只能视为开发期路径。生产安装、完整性校验、更新/回滚和扩展安装方式已拆成 [确定 OpenCLI Fork 的离线内置分发与能力握手](https://github.com/FrankQDWang/SeekTalent/issues/298)。能力握手仍然必需，因为离线内置不能阻止用户同时启用错误的上游扩展。

## 可重复验证

Fork 原型至少要有以下自动测试和 Chrome 集成验证：

1. 有一个已登录匹配 tab：在同一 `windowId` 创建 `active: false` 新 tab；原 active tab、窗口焦点和锚点 URL 不变。
2. 没有匹配 tab：返回 `host_tab_not_found`；tab 数和窗口数都不变。
3. 有多个候选：全部只读返回，不擅自选择或激活。
4. 用户 tab 与 owned tab 同窗：关闭 owned tab 后，用户 tab 和窗口仍存在且未导航。
5. owned tab 是该窗口中唯一的 SeekTalent tab：真实消失，不出现 `about:blank`。
6. 重复关闭：第二次返回 `already_missing`，仍是成功。
7. 人为制造 remove 失败：返回 `failed`，ownership 证据仍在，其他 tab 的 cleanup 继续。
8. 每个 tab 使用独立 session：任一命令只重置对应 tab 的 60 秒 deadline。
9. service worker 在倒计时中重启：按持久化绝对 deadline 回收，不重新获得 60 秒。
10. 用户通过 Chrome UI 手动关闭：lease registry 被清理，run 将其视作已回收。
11. tab 被拖到另一用户窗口：后续仍按精确 tab 身份工作，不把 tab 移回，也不关闭任一窗口。
12. 连接上游 1.8.6 扩展或缺少 capability：SeekTalent 启动前明确失败，绝不回退到 `chrome.windows.create` 路径。

原型应固定到上述 commit，并记录 fork commit、CLI 构建产物 hash、扩展构建产物 hash 和 Chrome 版本，保证结果可重放。

## 最终边界

薄 fork 只回答“在用户窗口旁边安全创建一个自己拥有的后台 tab，并能可靠关闭它”。SeekTalent 回答“为什么创建、属于哪个 source run、页面显示什么、何时立即结束、失败如何告警”。这个边界既满足当前需求，也避免把猎聘或 Dokobot 风格产品逻辑塞进 OpenCLI。
